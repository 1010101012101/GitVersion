Make Semantic Versioning easy

GitVersion uses your Git history to infer your semantic version using a set of built in conventions. It currently works with GitHubFlow and GitFlow.

Solves silly problems

GitVersion was created to solve problems in real projects, and while building it we learnt a lot..

Make use of those learnings and start asking meaningful questions rather than figuring out how to simply make your versioning work.

Flexible

GitVersion is used on a range of different projects from simple open source libraries to enterprise applications which need to support multiple versions, because of this you can use GitVersion in many ways including a command line tool, Ruby gem or MSBuild Task.
Also you have the option to pick the way you need your version number formatted, like NuGet packages using one format and assemblies using another.

Repeatable, Automated Versioning

Semantic Versioning has some really simple rules, by taking known branching strategies in git we can infer the version number of your product, and because the version is infered from your source control re-running builds will give you consistent build numbers.

Enables Continuous Delivery

One of the goals of GitVersion is to allow continuous delivery by knowing the upcoming version making it easy to produce packages which are candidates to be released.
